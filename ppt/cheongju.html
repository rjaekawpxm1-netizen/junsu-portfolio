청주 교통사고 분석 (RAG)

개인 프로젝트 · 분석·개발

분석·개발
GitHub Repository ↗

1프로젝트 개요

청주 교통사고 공개 데이터를 분석해 위험한 패턴을 찾았습니다.
그 결과를 그냥 질문하면 답이 나오게 RAG로 이어봤습니다.
분석이 보고서로 끝나지 않고, 누구나 물어볼 수 있게 하고 싶었습니다.

담당 역할

분석(EDA), 문서 정리, 벡터화, RAG 구성까지 혼자 했습니다.

2문제 정의 · 추진 배경

분석 결과가 전문가용 보고서로 끝나면, 정작 쓸 사람이 이해하기 어렵습니다.
그래서 결과를 그냥 물어보면 답이 나오게 만들 수 있을지 해봤습니다.

기존 방식의 한계

  • 분석이 보고서로 끝나 비전문가가 쓰기 어렵다.
  • 필요한 내용을 찾으려면 문서를 직접 읽어야 한다.
  • 통계만으로는 바로 판단에 쓰기 어렵다.
핵심 질문

“교통사고 분석 결과를 비전문가도 그냥 물어봐서 답을 받을 수 있을까?”

3데이터 · 입력 정보

  • 청주 교통사고 공개 데이터
  • 시간·지역·유형별 분석 결과 문서

분석 결과를 요약 문서로 정리해 벡터로 바꿨습니다.
질문이 들어오면 필요한 부분만 찾아 LLM이 답하게 했습니다.

4해결 접근 방식

  • EDA로 시간·지역·유형별 패턴과 위험 구간을 찾았다.
  • 분석 결과를 요약 문서로 정리하고 FAISS로 벡터화했다.
  • RAG로 질문에 필요한 부분만 찾아 자연어로 답하게 했다.

5문제 해결 과정

문제 1. 분석과 활용 사이가 끊김
문제 상황결과가 보고서에 머물러 비전문가가 쓰기 어려웠습니다.
해결 과정
  • 분석 결과를 요약 문서로 정리했다.
  • 물어보면 답이 나오게 해서 접근성을 높였다.
결과"야간 사고 주요 원인은?" 같은 질문이 가능해졌다.
문제 2. 필요한 내용만 정확히 찾기
문제 상황LLM이 문서 전체를 그냥 다루면 부정확하고 느렸습니다.
해결 과정
  • 문서를 FAISS로 벡터화했다.
  • RAG로 질문과 관련된 부분만 찾아 썼다.
결과필요한 부분만 찾아 더 정확하게 답했다.
문제 3. 다시 쓸 수 있는 구조로
문제 상황분석과 활용이 한 덩어리면 고치고 늘리기 어렵습니다.
해결 과정
  • analysis와 llm_rag 두 층으로 나눴다.
  • 결과가 바뀌면 검색 색인만 다시 만들게 했다.
결과분석과 활용을 나눠 늘리기 쉽게 만들었다.

6프로젝트 의의

  • 분석을 질문·답변 서비스로 이어봤다.
  • RAG로 분석 결과를 더 쉽게 쓰게 만들어봤다.
  • 분석과 활용을 나눠서 설계해봤다.
