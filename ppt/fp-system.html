1프로젝트 개요

제안요청서(RFP)를 올리면, LLM이 그 안의 기능을 뽑아 정리하고 개발비까지 계산해 주는 사내 도구입니다.
사람마다 다르던 산정 기준을 맞추고, 걸리는 시간을 줄이려고 만들었습니다.

담당 역할

기획, 화면, 서버, LLM 프롬프트, 검증 로직까지 혼자 만들었습니다.
지금 사내에서 실제로 쓰고 있습니다.

2문제 정의 · 추진 배경

원래는 담당자가 RFP를 직접 읽고 기능을 나누고 점수를 매겨 개발비까지 내야 했습니다.
이 과정이 2주쯤 걸렸고, 사람마다 결과도 달랐습니다.
이걸 매번 손으로 반복하지 않게 만들고 싶었습니다.

기존 방식의 한계

  • RFP를 사람이 다 읽고 기능을 나눠야 해서 2주쯤 걸린다.
  • 누가 하느냐에 따라 결과가 달라진다.
  • 계산을 손으로 하다 보니 실수가 난다.
핵심 질문

“RFP만 올리면 기능 정리부터 개발비 계산까지 한 번에 끝낼 수 있을까?”

3데이터 · 입력 정보

  • 제안요청서(RFP) 문서
  • 기능 분류·FP 산정 기준
  • 개발 유형별(신규/유지보수/재구축) 단가

RFP에서 글을 뽑아 LLM이 기능을 정리합니다.
그 결과를 계산·검증 로직이 받아서 FP와 개발비를 냅니다.

4해결 접근 방식

  • RFP에서 글을 뽑고, Claude로 기능을 정리한다(신규/고도화 구분).
  • 계산·보정·검증 로직을 따로 모듈로 만들고 테스트로 확인했다.
  • 신규·유지보수·재구축 개발비를 각각 내고, Supabase에 저장하고 엑셀로 내보낸다.

5문제 해결 과정

문제 1. 사람마다 결과가 달랐음
문제 상황같은 RFP라도 담당자에 따라 뽑는 기능과 점수가 달랐습니다.
해결 과정
  • LLM이 기능을 정리하도록 프롬프트를 정해뒀다.
  • 정해둔 규칙으로 점수를 매겼다.
결과누가 하든 결과가 비슷해졌다.
문제 2. LLM 결과를 그대로 믿기 어려움
문제 상황LLM이 그럴듯하지만 틀린 기능이나 숫자를 낼 때가 있었습니다.
해결 과정
  • LLM 결과를 계산·검증 로직으로 한 번 거른 뒤 썼다.
  • 테스트로 계산이 맞는지 확인했다.
결과LLM의 편함과 계산의 정확함을 같이 챙겼다.
문제 3. 산정에 너무 오래 걸림
문제 상황기능 나누고 점수 매기고 개발비 내는 데 2주쯤 걸렸습니다.
해결 과정
  • 올리면 분석부터 결과까지 한 번에 이어지게 만들었다.
  • 결과를 바로 엑셀로 받게 했다.
결과2주쯤 걸리던 일이 5분 정도로 줄었고, 사내에서 쓰고 있다.

6프로젝트 의의

  • 반복 업무를 LLM으로 자동화해 시간을 크게 줄여봤다.
  • LLM 결과를 검증 로직으로 걸러 실무에 쓸 수 있게 만들었다.
  • 기획부터 운영까지 혼자 끝까지 해봤다.
